# Plan 08-02: World Object Per-Spawn Loot & Trigger Config

## Goal
Allow each world object spawn to override:
1. The loot entry (which loot table drops when chest/object is opened) — per-spawn override
2. The trigger that fires on interaction — shown as read-only link to trigger editor in the UI

## Files to Modify

Add two optional fields to `ObjectSpawnEntry`:

## Spawn Loading Site — Confirmed Location
The actual spawn loading is in **`F:\mmo\src\shared\game_server\world\world_instance.cpp` lines 127–155** (NOT in `src/world_server/`).

The `WorldObjectSpawner` is constructed there. To pass `loot_entry` through, the fix requires **4 steps**:

1. **`world_object_spawner.h`** — add `uint32 m_lootEntryOverride = 0;` member; extend constructor to accept `uint32 lootEntryOverride = 0`
2. **`world_object_spawner.cpp`** — store it; in `SpawnOne()` add `spawned->SetLootEntryOverride(m_lootEntryOverride);` after `SpawnWorldObject` (around line 58)
3. **`world_instance.cpp` line 141** — pass `spawn.loot_entry()` as the new trailing argument to `WorldObjectSpawner` constructor
4. **`game_world_object_s.h/.cpp`** — add `m_lootEntryOverride` member + `SetLootEntryOverride()` method + use override in `OnUse`

> ⚠️ Objects are spawned **inside** the `WorldObjectSpawner` constructor — a post-construction setter on the spawner would be too late. The override must be injected at construction time and passed into `SpawnOne()`.

## Data Flow
1. Editor: user sets loot_entry=42 on a specific chest spawn → saved to `ObjectSpawnEntry.loot_entry`
2. Server loads map → creates `GameWorldObjectS` → calls `SetLootEntryOverride(42)`
3. Player opens chest → `OnUse` picks loot entry 42 (not the base objectlootentry)
4. If loot_entry==0 (default) → falls back to `m_entry.objectlootentry()` (existing behavior unchanged)
